The N+1 query problem occurs when code fetches a collection of records, then issues a separate query for each record to load related data — turning what should be a constant number of queries into one that scales linearly with collection size.

## The Mechanism

The issue is structural: an ORM or data access layer lazily loads associations, so iterating over results triggers hidden queries. You write what looks like simple iteration, but each loop body hits the database independently. The "N+1" name is literal — 1 query to fetch N records, then N queries to fetch their associations.

```python
# Looks harmless
users = User.all()           # 1 query
for user in users:
    print(user.posts.count)  # N queries — one per user
```

The database round-trip cost compounds: even if each query is fast (1ms), 500 users means 501 queries and ~500ms of pure network/parse overhead, plus you've made the DB do 500x the work it needed to.

## The Fix

Eager loading collapses N+1 into a predictable 2 queries (or 1 with a JOIN):

```python
users = User.includes(:posts).all()  # Rails: 2 queries
# or
users = User.joins(:posts).select(...)  # 1 query with JOIN
```

The ORM fetches all associated records in bulk and maps them in memory. The DB does one round-trip; your application does the association work.

## Where This Bites in Practice

**Backend services**: REST endpoints that serialize nested resources are the classic trap. A `/orders` endpoint that renders each order's line items triggers N queries per request. At low load it's invisible; at scale it's the first thing that falls over.

**GraphQL resolvers**: Each field resolver is independent by default. Without a DataLoader (batching layer), a `users { posts { title } }` query generates N queries for posts. DataLoader solves this by batching resolver calls within a single event loop tick.

**Fullstack with ORMs**: Next.js server components or Rails controllers that pass records to views/templates — the view iterates associations the controller didn't eagerly load. The template looks clean; the query log is not.

**Admin panels / reporting**: Autogenerated admin UIs (ActiveAdmin, Django Admin) are notorious for this. They render association counts or labels per-row with no awareness of query cost.

## How to Catch It

ORM query logging is the fastest signal — look for repeated queries with different IDs. The `bullet` gem in Rails or Django Debug Toolbar will surface these automatically. In production, query count per request is a useful metric to track; a sudden spike usually points here.

The mental model: if the number of queries you issue is a function of your dataset size, something is wrong.
